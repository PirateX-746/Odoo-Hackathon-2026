## Detailed Findings by Severity

### [P1] Error-boundary pages hard-code raw colors, breaking the theme system
- **Location**: `frontend/app/global-error.tsx:20-26`, `frontend/app/not-found.tsx:6-10`
- **Category**: Theming
- **Impact**: Every other surface in the app uses semantic tokens (`text-foreground`, `text-muted-foreground`, `bg-primary`, the `signal-*` scale) with dark as the default theme. These two pages instead hard-code `text-red-600`, `text-gray-500`, `text-gray-800`, `bg-blue-600`, `text-white` and skip the shadcn `Button` component for a raw `<button>`/`<a>`. In dark mode (the app default) these render with no relation to the surrounding palette, and if a user hits either page mid-session the visual language breaks completely — the one place users land after something has already gone wrong.
- **WCAG/Standard**: Not a hard contrast failure in isolation, but violates the app's own theming contract (1.4.1-adjacent consistency expectation) and 3.2.4 (Consistent Identification).
- **Recommendation**: Replace hard-coded classes with theme tokens (`text-foreground`, `text-muted-foreground`, `text-signal-critical`) and swap the raw `<button>`/`<a>` for the shared `Button` component so these states match the rest of the app.
- **Suggested command**: `/impeccable harden`

### [P1] Side-stripe border accent on dashboard alert list
- **Location**: `frontend/app/(main)/page.tsx:222-228`
- **Category**: Anti-Pattern
- **Impact**: `border-l-2` combined with `border-l-signal-critical` / `border-l-signal-warn` is the classic AI-slop "colored left border on a list item" tell — explicitly banned. It's a small visual detail but it's the one place in an otherwise clean design system where a default reflex slipped in.
- **Recommendation**: Rewrite using a leading icon/dot, a background tint, or a full border instead of a partial colored edge.
- **Suggested command**: `/impeccable polish`

### [P2] Form validation errors aren't programmatically linked to their inputs
- **Location**: Every react-hook-form usage — confirmed in `VehicleForm.tsx`, `DriverForm.tsx`, `TripForm.tsx`, `LoginForm.tsx`, `CompleteTripDialog.tsx` (same pattern repeats in Fuel/Expense/Maintenance/User forms since they follow the identical `register()` + adjacent `<p>` convention)
- **Category**: Accessibility
- **Impact**: Errors render as a sibling `<p className="text-xs text-destructive">` next to the input, but `register()` from react-hook-form doesn't set `aria-invalid` or `aria-describedby` on its own, and none of the forms add them manually. Sighted users see the red text; screen-reader users get no indication a field failed validation or what the error says, and `input.tsx`'s `aria-invalid:*` styling is dead code because the attribute is never actually set.
- **WCAG/Standard**: 3.3.1 (Error Identification), 4.1.2 (Name, Role, Value) — both AA.
- **Recommendation**: Add `aria-invalid={!!errors.field}` and `aria-describedby="field-error"` (matching the error `<p id>`) to each `Input`/`Controller`-driven field. Worth a small shared helper since the pattern is identical across ~8 forms.
- **Suggested command**: `/impeccable harden`

### [P2] Row-action icon buttons are cramped for touch
- **Location**: `frontend/app/(main)/vehicles/page.tsx:150-177`, `frontend/app/(main)/trips/page.tsx:188-217` (same pattern in drivers/fuel/maintenance/expenses lists)
- **Category**: Responsive Design
- **Impact**: View/edit/delete actions use `size="icon-sm"` (28px) with only `gap-1` (4px) between them. This technically clears WCAG 2.2's 24px AA minimum, but three adjacent 28px targets with 4px of spacing invite mis-taps on a touchscreen — the kind of table these fleet-ops users will realistically use on a tablet in a yard or cab.
- **Recommendation**: Either bump spacing between row actions on mobile, or collapse them into a single overflow menu below `md`.
- **Suggested command**: `/impeccable adapt`

### [P3] Dashboard KPI grid skips the tablet breakpoint
- **Location**: `frontend/app/(main)/page.tsx:101`
- **Category**: Responsive Design
- **Impact**: `grid-cols-2 gap-3 lg:grid-cols-4` jumps straight from 2 to 4 columns with no `md:` step, so ~768–1024px viewports (iPad-class tablets) get two oversized tiles instead of a proportionate 3- or 4-up layout.
- **Recommendation**: Add a `md:grid-cols-3` (or similar) intermediate step.
- **Suggested command**: `/impeccable adapt`

### [P3] Error/404 pages skip the app's component system
- **Location**: `frontend/app/global-error.tsx`, `frontend/app/not-found.tsx`
- **Category**: Anti-Pattern (consistency)
- **Impact**: Same two files as the P1 theming issue — they also use raw `<button>`/`<a>` markup instead of the shared `Button`, so even after retokenizing colors, the interactive elements won't match the rest of the app's hover/focus/active states.
- **Recommendation**: Bundle this into the same pass as the P1 token fix above.
- **Suggested command**: `/impeccable harden`

## Patterns & Systemic Issues

- **Form a11y gap is structural, not incidental.** Every form in the app is built on the same `register()` + adjacent error `<p>` convention, so the missing `aria-invalid`/`aria-describedby` wiring is a single fix applied ~8 times, not 8 separate bugs.
- **The two hard-coded-color files are the only theming gap in the whole app.** Every other one of the ~50 files sampled used semantic tokens exclusively — this is an isolated miss on Next.js's special-cased `global-error`/`not-found` routes, not a systemic habit.

## Positive Findings

- **Deliberate, well-documented token system**: OKLCH throughout, dark set as the default theme with `.light` as explicit opt-in, and a dedicated `signal-ok/warn/critical/idle` scale shared by one `StatusBadge` across Vehicle/Driver/Trip/Maintenance statuses — code comments explain *why* (`globals.css:43`, `StatusBadge.tsx:7-9`).
- **Genuine typographic contrast pairing**: Poppins for UI text, IBM Plex Mono reserved specifically for numeric/data readouts (VINs, plates, odometer, timestamps) — a documented, intentional choice (`layout.tsx:12-13`), not two similar sans-serifs colliding.
- **Tables handle overflow correctly**: every `Table` wraps in its own `overflow-x-auto` container (`table.tsx:9-11`) rather than letting wide rows blow out the page — exactly the right pattern for dense fleet data on narrow viewports.
- **Debounced, referentially-stable list fetching**: 300ms search debounce on every list page, plus an explicit code comment in `usePaginatedList.ts:20-22` warning against passing an inline fetcher (would cause a refetch loop) — shows real engineering care, not just working-by-luck.
- **Role-gating is consistent and traceable**: `CAN_MANAGE_*`/`CAN_DELETE_*` constants are applied uniformly across every list/detail/form view, and `constant.tsx` comments explicitly cross-reference the backend's `@Roles()` decorators so the two stay in sync.
